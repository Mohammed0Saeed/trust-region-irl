  Why there's no checkpoint

  Saving only happens at evaluation_and_save_frequency (trirl_ppo.py:700–702). Your run script uses the default
  evaluation_and_save_frequency = -1, which resolves to saving only at the very end of total_timesteps=30e6 (line
  74–75). All four runs are from today (10:54–12:25) — the NaN-diverging runs we just diagnosed — and none reached the
  end, so nothing was written. So: you need to retrain to completion (now with the IK/reward fixes) before there's a
  policy to view. Optionally pass --algorithm.evaluation_and_save_frequency=<steps> so it checkpoints periodically
  instead of only at the end.

  How visualization works (already built in)

  The mechanism is fully there — no new code needed. The pieces:

  - --runner.mode="test" → runner._test() calls model.test() (trirl_ppo.py:798), which rolls out the deterministic
  policy (action = action_mean, line 826) and calls self.train_env.render(env_state) each step (836–837).
  - --environment.render=True → FrankaPushT.__init__ opens the GLFW MujocoViewer (environment.py:157–159); render()
  converts the mjx batched state to a single mj_data for env 0 and draws it (environment.py:404–407).
  - --runner.load_model=<path>/latest.model → TRIRL_PPO.load() (trirl_ppo.py:770) unzips the checkpoint and restores
  policy/critic.
  - Test mode forbids wandb, tensorboard, and save_model (runner.py:358–363) — you must turn those off.
